2. SYSTEM DESCRIPTION
---------------------

* Overall Architecture:
  The system utilizes a decoupled, asynchronous micro-architecture. A high-
  performance React frontend captures real-time candidate biometric data (audio/
  screen focus) and communicates via REST/WebSockets with a Python API. The 
  backend acts as an orchestrator, piping audio to external STT services, 
  managing session persistence in a NoSQL database, and executing prompt chains 
  against large language models for evaluation.

* Technologies Used (Detailed Tech Stack):
  - Frontend Layer: Next.js 14 (App Router) & React 18 for server-side and 
    client-side rendering. Tailwind CSS for utility-first styling (utilizing 
    a 'Glassmorphism' modern aesthetic). Framer Motion for micro-animations 
    (e.g., audio waveforms) to reduce cognitive load and candidate anxiety.
  - Backend Layer: FastAPI (Python 3.11+) optimized with Uvicorn. Selected for 
    its native async/await support, which is critical when handling long-
    polling AI API calls and concurrent audio file uploads without blocking. 
    Pydantic is used for strict runtime data validation.
  - AI / Machine Learning Layer: Azure OpenAI Ecosystem. GPT-4o-mini is 
    utilized for the core logic layer (resume parsing, adaptive question 
    generation, and rubric-based evaluation). Azure Whisper API handles 
    highly accurate, real-time Speech-to-Text (STT) transcription.
  - Database Layer: MongoDB Atlas with the Motor (async) Python driver. A 
    NoSQL document store allows tracking of unstructured array data like 
    interview transcripts, fluctuating rubric scores, and session states without 
    complex database migrations.
  - Security & Authentication: Argon2 for state-of-the-art password hashing. 
    python-jose for JWT (JSON Web Token) creation, managed primarily via secure, 
    HTTP-Only cookies to prevent XSS vulnerabilities.

* Workflow: User Input to Output:
  1. Ingestion: Candidate uploads a PDF resume. The system extracts text using 
     pdfplumber and sends it to GPT-4o to build a structured Knowledge Graph.
  2. Interaction Initialization: The browser requests Microphone and Fullscreen 
     permissions, establishing the proctoring environment.
  3. The Interview Loop:
     - AI asks a question (via Text-to-Speech).
     - Candidate answers using their microphone; frontend records the Blob.
     - Audio chunk is sent to the backend, transcribed by Whisper.
     - The AI Engine evaluates the text. If the answer is too vague, the "Smart 
       Retry" module asks a clarifying follow-up question.
  4. Final Output: Session closes, and the Report Engine compiles an aggregated 
     markdown/JSON report with rubric scores (1-10) for HR review.


3. MODULE DESCRIPTION (DEEP DIVE)
---------------------------------

* Frontend Modules:
  - Interview Engine (`interview-chat.tsx`): The flagship component. It leverages 
    the Web Audio API to capture microphone streams and render a live visual 
    audio waveform. It handles the UI state machine (Listening, Processing, 
    Speaking) and utilizes the browser's SpeechSynthesis API to vocalize AI prompts.
  - Proctoring Monitor (`proctoring-monitor.tsx`): A silent guardian component 
    that hooks into the browser's native `visibilitychange` and `fullscreenchange` 
    events. It immediately detects if a candidate switches tabs or exits full 
    screen, visually warns them, and logs the penalty to the backend.
  - HR Dashboard (`app/dashboard`): A secure routing group utilizing Server 
    Components to fetch candidate analytics, review interview transcripts, and 
    configure custom role-based questionnaires.

* Backend Modules:
  - AI Engine (`ai_engine.py`): The highest-complexity module. It defines the 
    system prompts and controls the "5-5-5 Question Pattern":
      a. Resume Verification Stage (Deep dives into claimed projects)
      b. Role Alignment Stage (System design / theoretical fit)
      c. HR/Behavioral Stage (STAR method soft-skills assessment)
  - API Router (`interview_routes.py`): Exposes RESTful endpoints for the frontend. 
    Manages session persistence so candidates can resume interviews if dropped.
  - Validation Service (`security.py`): Intercepts incoming requests, validates 
    the JWT bearer token, and ensures candidates cannot access HR-only routes.

* Database Module:
  - `users` Collection: Stores emails, hashed passwords, and RBAC roles.
  - `sessions` Collection: Ephemeral documents tracking the active question number, 
    current stage, and accumulated warnings to allow state recovery.
  - `interviews` Collection: Historical records containing the full Q&A transcript, 
    detected skills list, and the final 1-10 scores across various competencies.

* Post-Interview Report Module:
  - Report Engine (`report_engine.py`): An offline/background processor that 
    collates the raw transcript and uses the LLM to generate a synthesized 
    "Executive Summary" of the candidate's strengths and weaknesses for recruiters.
