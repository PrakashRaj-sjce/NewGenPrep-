5. FIX 4 - PASSWORD RESET FLOW
================================================================================

  PROBLEM:
    frontend/app/forgot-password/page.tsx exists but only does console.log
    No backend endpoint exists for password reset

  SOLUTION:
    Add backend endpoints for reset token generation and password update.
    Frontend already has the UI, just needs to call real API.

  FLOW:
    User clicks Forgot Password
         |
         v
    Enters email, clicks Send Reset Link
         |
         v
    POST /api/auth/forgot-password
         |
         v
    Backend checks if email exists in db.users
         |
         +---> NOT FOUND -> Return success anyway (security: no email enum)
         |
         +---> FOUND -> Generate 6-digit OTP code
                  |
                  v
               Store OTP + expiry (15 min) in db.password_resets
                  |
                  v
               Send OTP via email (notification_service)
                  |
                  v
               Return success message

    User receives OTP email
         |
         v
    Enters OTP + new password on reset page
         |
         v
    POST /api/auth/reset-password
         |
         v
    Backend validates OTP and expiry
         |
         +---> INVALID or EXPIRED -> Return error
         |
         +---> VALID -> Hash new password, update db.users
                  |
                  v
               Delete used OTP from db.password_resets
                  |
                  v
               Return success, redirect to login

  FILES TO MODIFY:
    frontend/app/forgot-password/page.tsx -> Add OTP input and API calls
  FILES TO CREATE:
    backend/routes/auth_routes.py -> forgot-password and reset-password

8. FIX 7 - RATE LIMITING AND API SECURITY
================================================================================

  PROBLEM:
    No rate limiting on any endpoint.
    A single bad actor can flood the API.
    At 1000-3000 concurrent users, we need protection.

  SOLUTION:
    Add slowapi rate limiting middleware to FastAPI.
    Different limits per endpoint category.

  RATE LIMITS:
    Endpoint Category          | Limit
    ---------------------------|------------------
    /api/auth/register         | 5 per minute per IP
    /api/auth/login            | 10 per minute per IP
    /api/auth/forgot-password  | 3 per minute per IP
    /api/interview/respond     | 30 per minute per user
    /api/interview/upload-*    | 10 per minute per user
    /api/hr/*                  | 60 per minute per user
    /api/practice/*            | 20 per minute per user
    /api/transcribe            | 30 per minute per user
    /health                    | No limit

  FILES TO MODIFY:
    backend/interview_api.py -> Add slowapi middleware
    backend/requirements.txt -> Add slowapi dependency

10. FIX 9 - CONNECTION POOLING
================================================================================

  PROBLEM:
    AsyncIOMotorClient is created with default pool settings.
    At 3000 concurrent users, the default pool (100 connections) may exhaust.

  SOLUTION:
    Configure Motor client with explicit pool settings.
    Set maxPoolSize, minPoolSize, and connection timeouts.

  CONFIGURATION:
    maxPoolSize = 200       (handles burst of 3000 users)
    minPoolSize = 10        (keeps warm connections)
    maxIdleTimeMS = 30000   (close idle after 30s)
    connectTimeoutMS = 5000 (fail fast on connection issues)
    serverSelectionTimeoutMS = 5000

  FILES TO MODIFY:
    backend/interview_api.py -> Update AsyncIOMotorClient constructor
    (or new db_service.py after refactor)

11. ARCHITECTURE LAYER DIAGRAM (AFTER ALL FIXES)
================================================================================

  REQUEST FLOW (1000-3000 concurrent users):

    User Browser / Extension / Mobile
         |
         v
    [NGINX / Load Balancer] (optional, for production)
         |
         v
    [FastAPI Application]
         |
         +---> Rate Limiter (slowapi) checks IP/user limits
         |        |
         |        +---> REJECTED (429 Too Many Requests)
         |        |
         |        v

         |        |

         |        |

         |        |

                  |

                  |
                  +---> Azure OpenAI (AI question/evaluation)
                  |
                  +---> SMTP (email notifications)
                  |
                  v
               Response returned to client


  DATA FLOW FOR INTERVIEW SESSION:

    Candidate uploads resume (PDF)
         |
         v
    pdfplumber extracts text
         |
         v
    Azure GPT parses into Knowledge Graph
         |
         v
    Session created in MongoDB (indexed by session_id, user_id)
         |
         v
    Interview starts -> 15 questions generated (5-5-5 pattern)
         |
         v
    For each question:
         |
         +---> AI generates question (GPT)
         |
         +---> Candidate speaks answer
         |
         +---> Audio transcribed (Whisper)
         |
         +---> Answer evaluated (GPT) with rubric scoring
         |
         +---> If score < threshold -> Clarification question (max 2)
         |
         +---> Session updated in MongoDB
         |
         v
    Interview completes
         |
         v
    Report generated (full breakdown per question, stage, competency)
         |
         v
    HR notified (email)
