1. SYSTEM OVERVIEW
================================================================================

  The AI Interview Bot is an enterprise recruitment platform that automates
  first-round technical and behavioral screening using Generative AI.

  Three client interfaces feed into a single backend:
    1. Web Frontend (Next.js) - Primary candidate and HR interface
    2. Chrome Extension - Recruiter shortcut for quick scheduling
    3. Docker deployment - Production containerized setup

  The backend is a FastAPI (Python) server that orchestrates:
    - User authentication (JWT + Argon2)
    - AI-powered interview sessions (Azure OpenAI GPT + Whisper)
    - Real-time proctoring enforcement
    - Rubric-based scoring
    - Email notifications
    - Interview scheduling with automated reminders

6. DATABASE SCHEMA
================================================================================

  COLLECTION: users
    {
      id:          string (UUID, unique)
      email:       string (unique, indexed)
      password:    string (Argon2 hash)
      name:        string
      role:        string (candidate | hr | admin)
      created_at:  string (ISO datetime)
    }

  COLLECTION: sessions
    {
      session_id:          string (UUID, unique, indexed)
      user_id:             string (indexed)
      status:              string (created | resume_uploaded | in_progress |
                                   completed | terminated) (indexed)
      resume_filename:     string
      resume_analysis:     object (AI-parsed knowledge graph)
      target_company:      string (optional)
      question_count:      number
      total_questions:     number (default 15)
      current_stage:       number (1=resume, 2=role, 3=hr)
      clarification_count: number
      questions_asked:     array of question objects
      responses:           array of response objects with scores
      conversation_history: array of role/content pairs
      answers:             array
      warning_count:       number
      incidents:           array of proctoring incidents
      assigned_rubric_id:  string (optional)
      report:              object (generated on completion)
      created_at:          string (ISO datetime, indexed)
      completed_at:        string (ISO datetime)
    }

  COLLECTION: schedules
    {
      id:               string (UUID, unique, indexed)
      candidateName:    string
      candidateEmail:   string (indexed)
      date:             string (YYYY-MM-DD, indexed)
      time:             string (HH:MM)
      type:             string (technical | behavioral | final)
      duration:         number (minutes)
      role:             string (optional)
      status:           string (scheduled | sent | confirmed | cancelled)
      timezone:         string
      notes:            string
      reminded_24h:     boolean
      reminded_1h:      boolean
      notificationsSent: array
      createdBy:        string
      created_at:       string
    }

  COLLECTION: question_sets
    {
      id:             string (UUID, unique, indexed)
      category:       string (indexed)
      domain:         string
      questions:      array of {text, id, usageCount, lastUsedAt}
      questionCount:  number
      isActive:       boolean (indexed)
      isMandatory:    boolean
      uploadedBy:     string
      createdAt:      string
      updatedAt:      string
    }

  COLLECTION: rubrics
    {
      id:              string (UUID, unique, indexed)
      name:            string
      description:     string
      competencies:    array of {name, weight, description}
      totalQuestions:   number (min 15)
      active:          boolean
      created_at:      string
    }

  COLLECTION: settings
    {
      type:               string (unique, e.g. hr_config)
      max_questions:      number
      adaptive_difficulty: boolean
      voice_input:        boolean
      updated_at:         string
      updated_by:         string
    }

  COLLECTION: password_resets (NEW)
    {
      email:       string (indexed)
      otp_code:    string (hashed)
      expires_at:  datetime (TTL index, auto-delete after expiry)
      created_at:  datetime
    }

8. DATA FLOW DIAGRAMS
================================================================================

  INTERVIEW PIPELINE (Complete Session Flow):

    [Candidate]
         |
         | 1. Upload Resume (PDF, max 5MB)
         v
    [POST /api/interview/upload-resume]
         |
         | 2. pdfplumber extracts text
         v
    [Azure GPT parses resume into Knowledge Graph]
         |
         | 3. Returns session_id + analysis
         v
    [POST /api/interview/start]
         |
         | 4. Generates opening question from resume skills
         v
    [Interview Loop - 15 iterations]
         |
         +----> Candidate speaks answer
         |           |
         |           v
         |      [Azure Whisper transcribes audio]
         |           |
         |           v
         |      [POST /api/interview/respond]
         |           |
         |           v
         |      [Azure GPT evaluates answer]
         |           |
         |           +---> Quality Score > threshold?
         |           |        |
         |           |        +--- YES: Generate next question
         |           |        |         (5-5-5 pattern selection)
         |           |        |
         |           |        +--- NO:  Clarification? (max 2)
         |           |                   |
         |           |                   +-- YES: Generate follow-up
         |           |                   |
         |           |                   +-- NO:  Accept and move on
         |           |
         |           v
         |      [Save to MongoDB]
         |           |
         +----<------+  (repeat for 15 questions)
         |
         v
    [Interview Complete]
         |
         | 5. Generate report (per-question breakdown)
         v
    [Save report to session document]
         |
         | 6. Notify HR via email
         v
    [Done]


  PROCTORING ENFORCEMENT FLOW:

    [Interview Active]
         |
         +---> [Fullscreen Lock Active]
         |        |
         |        +---> User exits fullscreen?
         |                 |
         |                 +---> POST /api/interview/{id}/warning
         |                 |     type: fullscreen_exit
         |                 |
         |                 +---> Warning count incremented
         |                 |
         |                 +---> Count >= 3?
         |                        |
         |                        +--- YES: Auto-terminate
         |                        |         POST /api/interview/terminate
         |                        |
         |                        +--- NO:  Show warning overlay
         |
         +---> [Tab Switch Detector]
         |        |
         |        +---> User switches tab?
         |                 |
         |                 +---> Same flow as fullscreen exit
         |
         +---> [Copy-Paste Blocker]
                  |
                  +---> Paste event detected?
                           |
                           +---> Event prevented
                           +---> Warning logged silently

9. SECURITY ARCHITECTURE
================================================================================

  AUTHENTICATION:
    - Passwords hashed with Argon2 (memory-hard, GPU-resistant)
    - JWT tokens with 7-day expiry
    - Token refresh not implemented (candidate sessions are short-lived)
    - HR endpoints require role = hr in JWT payload

  API SECURITY:
    - CORS whitelist (localhost:3000, extension origin, production URL)
    - Rate limiting per IP and per user (slowapi)
    - File upload size limits (resume 5MB, recording 500MB)
    - Input sanitization on all user-provided strings
    - No SQL injection risk (MongoDB parameterized queries)

  PROCTORING SECURITY:
    - Fullscreen enforcement (browser API)
    - Tab switch detection (visibilitychange event)
    - Copy-paste interception (clipboard event listeners)
    - All violations logged to session.incidents[]
    - Auto-termination after configurable violation threshold

  DATA PRIVACY:
    - Resume data encrypted at rest (MongoDB Atlas encryption)
    - All traffic over HTTPS in production
    - Session isolation (user can only access own sessions)
    - AI processing ignores demographic markers (ethical AI)
    - Password reset OTPs auto-expire via TTL index

10. SCALABILITY DESIGN (1000-3000+ USERS)
================================================================================

  DATABASE LAYER:
    - MongoDB Atlas auto-scaling cluster
    - Indexed queries on all hot paths (session_id, user_id, email)
    - Connection pooling: 200 max connections, 10 min warm
    - TTL indexes for auto-cleanup of expired data

  APPLICATION LAYER:
    - FastAPI async handlers (non-blocking IO)
    - Rate limiting prevents abuse (slowapi)
    - AI calls use httpx with 60s timeout (prevent hanging)
    - Session data stored in DB (not in-memory for multi-instance)

  DEPLOYMENT LAYER:
    - Docker Compose for single-server deployment
    - For 3000+ users: Add NGINX reverse proxy with load balancing
    - Horizontal scaling: Run multiple FastAPI instances behind NGINX
    - Each instance connects to same MongoDB Atlas cluster

  BOTTLENECK ANALYSIS:
    Component          | Bottleneck Risk | Mitigation
    -------------------|-----------------|----------------------------------
    MongoDB queries    | MEDIUM          | Indexes + connection pooling
    Azure OpenAI calls | HIGH            | Rate limits + caching + fallback
    File uploads       | LOW             | Size limits + streaming
    WebSocket (future) | MEDIUM          | Horizontal scaling
    Email sending      | LOW             | Async + queue (future)
